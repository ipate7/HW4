Q1: FoundationDB decouples transaction processing from storage so each layer can scale and recover independently. The Transaction Management System handles read versions, commit ordering, conflict checking, and logging coordination, while the Storage System is responsible for actually storing key-value data and serving reads. This separation makes the system more modular and lets FoundationDB optimize control-plane work differently from data-plane work.

Q2: Read-write conflict detection is handled by the Resolver in the transaction management layer. The check happens during the commit process after the transaction has been assigned a commit version but before the commit is fully accepted. At that point, the Resolver compares the transaction’s read conflict ranges against writes from other recently committed transactions.

Q3: The read version gives a transaction a consistent snapshot of the database to read from. All reads in that transaction are interpreted as if they happened at that single version, which is what gives FoundationDB strict transactional consistency for normal reads. This means the client sees a stable snapshot even while other writes may be committing concurrently.

Q4: Snapshot reads do not add read conflict ranges, so they avoid the normal read-write conflict detection used for standard reads. Because of that, they can read data without causing the transaction to abort if another transaction later writes that same key or range. You would choose snapshot reads when you want better performance or non-blocking observation of data and do not need that read to participate in serializable conflict checking.

Q5: Recovery is cheap because LogServers durably store committed mutations during normal operation, while StorageServers apply those updates in the background. Since the logs already contain the committed state history needed to catch up, the system does not need to replay a traditional redo log in the critical path before becoming available again. A recovering StorageServer can rebuild state from the logs while the transaction system continues operating.
